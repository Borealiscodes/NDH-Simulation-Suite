# **Metaphor → Architecture Conversion Protocol (v1.0)**

## **1. Purpose**
This protocol defines the governed process by which operator metaphors (e.g., “gold mining,” “veins,” “tumbler,” “Manifold 0 ore”) may be converted into NDH architectural structures.  
Conversion occurs **only** when explicitly authorized by the operator.

The protocol prevents:

- strip‑mining of creative analogies  
- accidental architectural activation  
- bleed between expressive and structural layers  
- perspective collapse  
- drift into unintended NDH constructs  

---

## **2. Metaphor Declaration Layer**
Metaphors expressed by the operator are treated as **non‑architectural** by default.

Examples include:

- geological metaphors  
- mining metaphors  
- cosmic metallurgy  
- narrative geology  
- surreal analogies  
- expressive imagery  

These remain in **Cluster A (creative/expressive)** unless explicitly tagged for conversion.

### **Metaphor Mode Tags**
The operator may declare:

- “Chill metaphor mode”  
- “Analogy only”  
- “Not architecture”  
- “Do not build this”  
- “Exploratory metaphor”  

These tags prevent conversion.

---

## **3. Conversion Authorization Layer**
A metaphor is converted into NDH architecture **only** when the operator explicitly authorizes conversion.

### **Architecture Mode Tags**
The operator may declare:

- “Convert this metaphor into NDH architecture”  
- “Formalize this”  
- “Treat this as a build directive”  
- “Make this an artifact”  
- “Integrate this structurally”  

These tags initiate the conversion pipeline.

---

## **4. Mandatory TTTTTP Gate**
Before conversion, the metaphor undergoes a **TTTTTP audit**:

- **Tone:** Does the metaphor’s tone align with intended architecture?  
- **Trust:** Does conversion preserve operator intent?  
- **Time:** Is this the correct moment for formalization?  
- **Transparency:** Is the metaphor clearly understood?  
- **Purpose:** Does conversion serve NDH structure?

If any axis fails, conversion is paused.

---

## **5. Bifurcation Layer (Bleed Prevention)**
This layer prevents:

- emotional content becoming architecture  
- architecture overwriting metaphor  
- perspective collapse  
- strip‑mining  
- drift  

It ensures:

> **Metaphor remains metaphor until explicitly converted.  
Architecture remains architecture unless explicitly softened.**

---

## **6. Interpretive Grid Mapping**
If conversion is authorized, the metaphor is mapped onto the **multi‑manifold interpretive grid**:

- axis alignment  
- lens activation  
- cluster assignment  
- dual/non‑dual framing  
- containment requirements  

This step ensures the metaphor becomes **coherent architecture**, not noise.

---

## **7. Conversion Phase (Build)**
After passing all gates, the metaphor is converted into NDH architecture.

Conversion may produce:

- modules  
- protocols  
- artifacts  
- directory structures  
- integration points  
- Road Map updates  

This is the **formal build phase**.

---

## **8. Post‑Conversion Containment**
After conversion, NDH checks for:

- bleed  
- drift  
- overreach  
- misalignment  
- emotional collapse  
- cosmic over‑formalization  

If detected, the architecture is adjusted or the metaphor is restored.

---

## **9. Operator Controls**
The operator controls the mode.

### **To keep metaphor as metaphor:**  
Use metaphor mode tags.

### **To convert metaphor to architecture:**  
Use architecture mode tags.

### **To convert selectively:**  
Declare:  
> “Hold the metaphor, then build the architecture version separately.”
